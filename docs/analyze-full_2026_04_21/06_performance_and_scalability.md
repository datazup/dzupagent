# Performance and Scalability Review - dzupagent (2026-04-21)

This document is a static performance/scalability review of `dzupagent`. No profiling traces, load-test outputs, or production telemetry were used. Performance risk statements are explicitly marked as **[Inference]** when they come from code structure rather than measured runtime behavior.

## Repository Overview
- `dzupagent` is a large Yarn 1 monorepo with major runtime surfaces in server orchestration, queue/worker execution, memory analytics, and a Vue playground UI. Current local tree signals: `29` package manifests, `2647` TS/TSX/Vue source files, `1047` test files, and `67` server route files.
- Secondary artifacts align with this scale and workload profile: `2283` source files, `970` test files, and explicit queue + memory/retrieval signals in repo summaries ([DZUPAGENT.md:37](/media/ninel/Second/code/datazup/ai-internal-dev/out/workspace-repo-docs-static-portable-markdown/DZUPAGENT.md:37), [DZUPAGENT.md:38](/media/ninel/Second/code/datazup/ai-internal-dev/out/workspace-repo-docs-static-portable-markdown/DZUPAGENT.md:38), [dzupagent.md:15](/media/ninel/Second/code/datazup/ai-internal-dev/out/code-features-live-sample/summaries/repos/dzupagent.md:15), [dzupagent.md:69](/media/ninel/Second/code/datazup/ai-internal-dev/out/code-features-live-sample/summaries/repos/dzupagent.md:69)).
- Dominant review targets were run lifecycle paths, queue growth/retry behavior, SSE/WS event streaming and polling, memory analytics/export flows, and large-list rendering/data-fetch paths.

## Performance Profile
- **Run execution path** is the primary backend hot path: queue dequeue, repeated run state/log writes, optional trace writes, optional reflection/analyzer/context-transfer branches, and event emission all in a single completion pipeline ([run-worker.ts:249](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/runtime/run-worker.ts:249), [run-worker.ts:416](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/runtime/run-worker.ts:416), [run-worker.ts:447](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/runtime/run-worker.ts:447), [run-worker.ts:455](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/runtime/run-worker.ts:455), [run-worker.ts:602](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/runtime/run-worker.ts:602)).
- **Run observability/read path** can be heavy: `/api/runs/:id/logs` and `/api/runs/:id/trace` pull full per-run log history and then derive `toolCalls` and `phases` in-process ([runs.ts:452](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:452), [runs.ts:458](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:458), [runs.ts:463](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:463), [runs.ts:487](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:487), [postgres-stores.ts:136](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/postgres-stores.ts:136)).
- **Realtime path** combines bus fanout, SSE heartbeat/keepalive timers, and polling fallback loops per stream connection ([runs.ts:632](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:632), [events.ts:47](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/events.ts:47), [events.ts:63](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/events.ts:63), [sse-streaming-adapter.ts:100](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/streaming/sse-streaming-adapter.ts:100)).

## Performance Findings
1. **High - [Inference] Unbounded in-process queue and dead-letter growth risk in run execution paths.**  
Evidence: in-memory queue stores `pending` and `deadLetterQueue` as unbounded arrays with no max depth control ([run-queue.ts:89](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/queue/run-queue.ts:89), [run-queue.ts:96](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/queue/run-queue.ts:96), [run-queue.ts:227](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/queue/run-queue.ts:227)); BullMQ adapter also retains local dead-letter entries in memory ([bullmq-run-queue.ts:72](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/queue/bullmq-run-queue.ts:72), [bullmq-run-queue.ts:210](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/queue/bullmq-run-queue.ts:210)).  
Impact: memory footprint can scale with backlog/failure volume.

2. **High - [Inference] Run logs and trace endpoints are unpaginated and likely become latency/payload hotspots on verbose runs.**  
Evidence: `/api/runs/:id/logs` and `/api/runs/:id/trace` fetch full logs, and trace response also derives additional arrays from the full set ([runs.ts:458](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:458), [runs.ts:469](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:469), [runs.ts:487](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:487)); store method has no limit in query path ([postgres-stores.ts:136](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/postgres-stores.ts:136)).  
Impact: p95 latency and response size likely grow with per-run log cardinality.

3. **High - [Inference] SSE realtime scaling is linear with connection count due per-connection timers and polling checks.**  
Evidence: per-stream `setInterval` polling for run terminal state every 2s ([runs.ts:632](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/runs.ts:632)); events stream heartbeat timer and `while (!closed)` wait loop per connection ([events.ts:47](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/events.ts:47), [events.ts:63](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/events.ts:63)); SSE adapter also keeps keepalive timer checks ([sse-streaming-adapter.ts:100](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/streaming/sse-streaming-adapter.ts:100)).  
Impact: timer overhead and `runStore.get` read pressure grow with concurrent viewers.

4. **High - [Inference] Memory analytics/export paths materialize large data repeatedly and include expensive joins/transforms.**  
Evidence: analytics routes export up to `10_000` rows per request ([memory.ts:132](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/memory.ts:132)); `exportFrame` loads full namespace for non-query mode then slices ([memory-service-ext.ts:198](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory-ipc/src/memory-service-ext.ts:198), [memory-service-ext.ts:200](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory-ipc/src/memory-service-ext.ts:200)); analytics results call `toArray()` (full materialization) ([duckdb-engine.ts:189](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory-ipc/src/analytics/duckdb-engine.ts:189)); duplicate detection uses self-join ([memory-analytics.ts:319](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory-ipc/src/analytics/memory-analytics.ts:319)); JSON transport adds full stringify/parse + base64 overhead ([mcp-memory-transport.ts:143](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory-ipc/src/mcp-memory-transport.ts:143), [mcp-memory-transport.ts:201](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory-ipc/src/mcp-memory-transport.ts:201)).  
Impact: CPU and memory costs likely rise sharply as memory cardinality grows.

5. **Medium - [Inference] Write amplification in trace/vector persistence can constrain throughput.**  
Evidence: each trace step append executes read + insert + update cycle ([drizzle-run-trace-store.ts:53](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/drizzle-run-trace-store.ts:53), [drizzle-run-trace-store.ts:58](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/drizzle-run-trace-store.ts:58), [drizzle-run-trace-store.ts:69](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/drizzle-run-trace-store.ts:69)); vector upsert loops one DB upsert per entry ([postgres-stores.ts:330](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/postgres-stores.ts:330), [postgres-stores.ts:334](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/postgres-stores.ts:334)).  
Impact: DB round-trip count scales linearly with step/vector volume.

6. **Medium - [Inference] `forge_run_logs` and run-list access paths appear index-light for dominant read patterns.**  
Evidence: run logs table schema has no explicit index declarations on `run_id`/`timestamp` in code-defined schema ([drizzle-schema.ts:73](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/drizzle-schema.ts:73), [drizzle-schema.ts:80](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/drizzle-schema.ts:80)); migration shown adds `forge_runs_status_idx` but no `forge_run_logs` index ([0002_run_status_halted.sql:34](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/drizzle/0002_run_status_halted.sql:34)); list path sorts runs by `startedAt` ([postgres-stores.ts:88](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/persistence/postgres-stores.ts:88)).  
Impact: read performance may degrade under large history tables.

7. **Medium - [Inference] Frontend polling fanout and status-contract drift can create unnecessary request churn.**  
Evidence: terminal set in chat store omits `halted` ([chat-store.ts:16](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/chat-store.ts:16)); fallback polling runs every 3s until terminal ([chat-store.ts:161](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/chat-store.ts:161), [chat-store.ts:214](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/chat-store.ts:214)); app also runs health polling and SSE fallback paths ([App.vue:141](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/App.vue:141), [App.vue:167](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/App.vue:167), [health-store.ts:85](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/health-store.ts:85)); memory analytics polling triggers six parallel calls ([useMemoryAnalytics.ts:212](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/composables/useMemoryAnalytics.ts:212)).  
Impact: avoidable backend load and slower UI stabilization under degraded conditions.

8. **Medium - [Inference] Memory browse/list paths can read more data than needed before slicing.**  
Evidence: no-search memory-browse path calls `memoryService.get(namespace, scope)` then slices in route ([memory-browse.ts:51](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/routes/memory-browse.ts:51)); memory service `get` delegates to store search without a limit in that branch ([memory-service.ts:182](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory/src/memory-service.ts:182)); in-memory base store search iterates all namespace entries before optional pagination ([store-factory.ts:98](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory/src/store-factory.ts:98), [store-factory.ts:130](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/memory/src/store-factory.ts:130)).  
Impact: request cost can scale with namespace size even when response size is capped.

9. **Medium - [Inference] Eval orchestration queue has no hard pending cap and uses per-active-run lease timers.**  
Evidence: pending queue represented by unbounded arrays/sets ([eval-orchestrator.ts:69](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/services/eval-orchestrator.ts:69), [eval-orchestrator.ts:70](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/services/eval-orchestrator.ts:70), [eval-orchestrator.ts:356](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/services/eval-orchestrator.ts:356)); per-active-run lease refresh interval exists ([eval-orchestrator.ts:77](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/services/eval-orchestrator.ts:77), [eval-orchestrator.ts:629](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/server/src/services/eval-orchestrator.ts:629)).  
Impact: backlog and timer churn can increase under heavy eval enqueue bursts.

10. **Low - [Inference] Large-view UI rendering is non-virtualized in several heavy panels.**  
Evidence: run detail renders trace events and logs directly via `v-for` ([RunDetailView.vue:247](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/views/RunDetailView.vue:247), [RunDetailView.vue:305](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/views/RunDetailView.vue:305)); memory analytics tab renders multiple potentially large result sets directly ([MemoryAnalyticsTab.vue:406](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/components/inspector/MemoryAnalyticsTab.vue:406), [MemoryAnalyticsTab.vue:659](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/components/inspector/MemoryAnalyticsTab.vue:659)); chat message history has no explicit cap ([chat-store.ts:20](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/chat-store.ts:20)).

11. **Low - [Inference] Compatibility fallback count path can explode read cost on older store implementations.**

## Frontend Responsiveness Review
- Run detail load is correctly parallelized (`run`, `logs`, `trace` together), which minimizes request waterfall latency but amplifies impact of large log/trace payloads ([run-store.ts:168](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/run-store.ts:168), [run-store.ts:169](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/run-store.ts:169)).
- Chat completion waits on WS-first + REST polling fallback. This is resilient, but status mismatch (`halted` absent in terminal set) can keep polling alive longer than necessary ([chat-store.ts:16](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/chat-store.ts:16), [chat-store.ts:214](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/chat-store.ts:214)).
- App shell starts WS and health polling at mount and adds SSE fallback on WS error; this improves continuity but can increase background request/event overhead in unstable networks ([App.vue:141](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/App.vue:141), [App.vue:167](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/App.vue:167), [health-store.ts:85](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/health-store.ts:85)).
- Memory analytics view auto-refreshes and fans out six API requests per cycle. This is useful for visibility but can stress both client and server if many tabs are open ([MemoryAnalyticsTab.vue:141](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/components/inspector/MemoryAnalyticsTab.vue:141), [MemoryAnalyticsTab.vue:143](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/components/inspector/MemoryAnalyticsTab.vue:143), [useMemoryAnalytics.ts:212](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/composables/useMemoryAnalytics.ts:212)).
- Buffer discipline is generally strong in realtime stores (`MAX_EVENT_LOG`, `MAX_EVENTS`), reducing unbounded client-memory risk on live channels ([ws-store.ts:29](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/ws-store.ts:29), [trace-store.ts:12](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/stores/trace-store.ts:12)).
- Large-list rendering is still non-virtualized in run detail and analytics tables/cards, so frame drops are plausible as data grows ([RunDetailView.vue:247](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/views/RunDetailView.vue:247), [MemoryAnalyticsTab.vue:659](/media/ninel/Second/code/datazup/ai-internal-dev/dzupagent/packages/playground/src/components/inspector/MemoryAnalyticsTab.vue:659)).

## Backend And Data-Path Review

## Scalability Constraints
- **Process-memory ceilings in non-durable paths:** in-memory run queue/dead-letter growth and unbounded chat message history can accumulate with workload/session duration.
- **Read-amplification ceiling:** full log/trace retrieval and polling-driven stream checks increase both DB/store reads and payload transfer as data and subscribers grow.
- **Timer-cardinality ceiling:** SSE heartbeat/keepalive loops, stream polling intervals, cron trigger timers, and eval lease timers all scale with active connections/jobs/triggers.
- **Batch path cost-growth:** memory analytics/export/import rely on full materialization and JSON/base64 transforms, creating superlinear practical cost under large datasets.
- **DB-roundtrip growth:** trace-step and vector upsert write patterns are per-item and can bottleneck before compute saturation.
- **Index-risk ceiling:** missing/unclear indexes on highest-volume log reads can shift hot paths from indexed to scan/sort-heavy as tables age.
- **Fallback-path pathological risk:** `Number.MAX_SAFE_INTEGER` list fallback in count bridging can explode for legacy stores.

## Recommended Optimization Plan
### Immediate Work (1-2 sprints)
1. Add pagination/cursor parameters to `/api/runs/:id/logs` and `/api/runs/:id/trace`, with compact default summaries for UI-first render.
2. Add hard queue guardrails for in-memory run queues (`maxPending`, `maxDeadLetter`, rejection/backpressure behavior, metrics).
3. Introduce explicit indexes for run-log read patterns (`run_id`, `timestamp`) and validate with explain plans against `getLogs`.
4. Fix frontend run terminal status parity (`halted`) and centralize status enum/shared helper to prevent future drift.
5. Consolidate frontend polling cadence with jitter/backoff and visibility-aware throttling, especially for analytics + health + chat overlap.
6. Add request budget controls to memory analytics endpoints (row ceilings per endpoint, concurrency limit, and per-request cost telemetry).
7. Replace full-memory browse path for no-search with paged store-side retrieval where available.
8. Instrument payload size, row count, and endpoint latency on logs/trace/analytics to move decisions from inference to measured SLO tuning.

### Structural Work (multi-sprint)
1. Move memory export/import and analytics to streaming/chunked pipelines rather than full in-memory materialization and base64 JSON transport.
2. Refactor run trace persistence to append in batches or reduce round trips (atomic increment strategy or buffered writer).
3. Implement batch vector upsert SQL (`VALUES (...)` multi-row upsert) to reduce per-entry DB calls.
4. Replace per-connection stream polling with event-driven terminal state closure where possible, keeping polling as degraded fallback only.
5. Introduce admission control and global budgets for SSE/WS subscribers to avoid linear connection growth failure modes.
6. Add queue-depth SLO enforcement for eval/run queues with bounded pending backlog and overload responses.
7. Rework trigger scheduling toward consolidated scheduler ticks or wheel-based timing for large trigger counts.
8. Build repeatable load tests for run lifecycle, stream fanout, and memory analytics/export to validate each optimization against p95/p99 and cost targets.
