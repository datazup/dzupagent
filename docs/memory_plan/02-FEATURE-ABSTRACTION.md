## 1. Problem Statement

Currently, a "Feature" in StarterForge is a monolithic entity that couples:
- **What** the feature does (description, requirements, user intent)
- **How** it's implemented (specific tech stack, specific code files)
- **Where** it's used (template assignments, project context)

This tight coupling means:
- You cannot generate the same "User Authentication" feature for both Vue+Express AND React+FastAPI without creating two completely independent features
- RAG retrieval only finds features matching the same tech stack
- Lessons learned from generating "Auth" in Vue don't inform generating "Auth" in React
- Templates are locked to a single tech stack because their features are

### What We Need

A clear separation between:

```
FeatureSpec (ABSTRACT)                    FeatureImplementation (CONCRETE)
├── name: "User Authentication"           ├── techStack: {vue3, express, prisma}
├── description: "JWT-based auth..."      ├── files: Record<string, string>
├── category: "auth"                      ├── quality: 85
├── requirements: [...]                   ├── testResults: {...}
├── clarificationAnswers: {...}           ├── apiContract: {...}
├── scope: [frontend, backend, db]        ├── featureSpecId: "spec-auth-123"
└── tags: ["auth", "jwt", "rbac"]        └── generatedAt: "2026-03-22"
```

## 2. Data Model Design

### 2.1 New Prisma Models

```prisma
/// Abstract feature specification — tech-stack agnostic.
/// One FeatureSpec can have multiple FeatureImplementation records,
/// each for a different tech stack.
model FeatureSpec {
  id                String   @id @default(cuid())
  featureId         String   /// kebab-case identifier (e.g., "user-authentication")
  name              String
  description       String   @db.Text
  category          String
  tags              String[]
  scope             String[] /// ["frontend", "backend", "database", "tests"]

  /// Structured requirements extracted from clarification
  requirements      Json     @default("{}")

  /// Canonical clarification answers (tech-stack agnostic parts)
  clarificationAnswers Json  @default("{}")

  tenantId          String?
  createdBy         String?
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  /// Relations
  implementations   FeatureImplementation[]
  tenant            Tenant?  @relation(fields: [tenantId], references: [id])

  @@unique([featureId, tenantId])
  @@index([category])
  @@index([tenantId])
}

/// Concrete implementation of a FeatureSpec for a specific tech stack.
/// This is what gets generated by the feature-generator pipeline.
model FeatureImplementation {
  id                String   @id @default(cuid())
  featureSpecId     String
  featureSpec       FeatureSpec @relation(fields: [featureSpecId], references: [id], onDelete: Cascade)

  /// Tech stack this implementation targets
  techStack         Json     /// TechStack object
  techStackKey      String   /// Normalized key: "vue3:express:prisma:tailwind:vitest"

  /// Generation outcome
  files             String[] /// File paths in overlay
  quality           Int      @default(0)
  status            String   @default("draft")
  version           String   @default("1.0.0")

  /// Generation metadata
  testResults       Json?
  apiContract       Json?
  promptVersionIds  Json?
  generationCost    Json?    /// Token usage, model costs

  /// Overlay files relation (reuse existing OverlayFile model)
  overlayFiles      OverlayFile[]

  tenantId          String?
  createdBy         String?
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  @@unique([featureSpecId, techStackKey])
  @@index([techStackKey])
  @@index([tenantId])
}
```

### 2.2 Tech Stack Key Generation

```typescript
/**
 * Generate a normalized, sortable key for a tech stack configuration.
 * Used as a unique identifier for implementations of the same feature spec.
 *
 * Example: { frontend: 'vue3', backend: 'express', ... } → "vue3:express:prisma:tailwind:vitest"
 */
export function techStackKey(ts: TechStack): string {
  return [
    ts.frontend,
    ts.backend,
    ts.database,
    ts.styling,
    ts.testing,
  ].join(':')
}
```

### 2.3 Relationship to Existing Feature Model

The existing `Feature` model in Prisma serves as the current implementation. Migration path:

```
PHASE 1 (Non-breaking):
  - Create FeatureSpec and FeatureImplementation models
  - Add optional featureSpecId to existing Feature model
  - New generations create FeatureSpec + FeatureImplementation
  - Existing features continue working as-is

PHASE 2 (Migration):
  - Backfill FeatureSpec records from existing Feature records
  - Link Feature.featureSpecId to corresponding FeatureSpec
  - Feature model becomes a thin wrapper delegating to FeatureImplementation

PHASE 3 (Cleanup — optional):
  - Deprecate direct Feature creation
  - All generation goes through FeatureSpec → FeatureImplementation
```

## 3. Service Layer Design

### 3.1 FeatureSpecService

```typescript
// apps/api/src/services/features/feature-spec.service.ts

export interface CreateFeatureSpecInput {
  name: string
  featureId: string
  description: string
  category: string
  tags: string[]
  scope: FeatureScope[]
  requirements?: Record<string, unknown>
  clarificationAnswers?: Record<string, string | string[]>
  tenantId: string
  userId: string
}

export const featureSpecService = {
  /**
   * Create or find an existing feature spec.
   * Two specs are considered identical if they share featureId + tenantId.
   * This enables "generate same feature for different stack" use case.
   */
  async findOrCreate(input: CreateFeatureSpecInput): Promise<FeatureSpec> {
    const existing = await prisma.featureSpec.findUnique({
      where: { featureId_tenantId: { featureId: input.featureId, tenantId: input.tenantId } },
    })
    if (existing) return existing

    return prisma.featureSpec.create({
      data: {
        featureId: input.featureId,
        name: input.name,
        description: input.description,
        category: input.category,
        tags: input.tags,
        scope: input.scope,
        requirements: input.requirements ?? {},
        clarificationAnswers: input.clarificationAnswers ?? {},
        tenantId: input.tenantId,
        createdBy: input.userId,
      },
    })
  },

  /**
   * Get all implementations of a feature spec.
   * Returns which tech stacks this feature has been generated for.
   */
  async getImplementations(specId: string): Promise<FeatureImplementation[]> {
    return prisma.featureImplementation.findMany({
      where: { featureSpecId: specId },
      include: { overlayFiles: true },
      orderBy: { quality: 'desc' },
    })
  },

  /**
   * Check if an implementation exists for a given tech stack.
   * Used to decide whether to generate or retrieve cached.
   */
  async hasImplementation(specId: string, techStack: TechStack): Promise<boolean> {
    const key = techStackKey(techStack)
    const count = await prisma.featureImplementation.count({
      where: { featureSpecId: specId, techStackKey: key },
    })
    return count > 0
  },

  /**
   * Find similar feature specs across all tenants (for marketplace/RAG).
   * Uses description similarity, not tech stack — that's the point.
   */
  async findSimilar(description: string, params: {
    tenantId: string
    category?: string
    limit?: number
  }): Promise<FeatureSpec[]> {
    // Phase 1: DB text search
    // Phase 2: Qdrant embedding search on FeatureSpec descriptions
    return prisma.featureSpec.findMany({
      where: {
        tenantId: params.tenantId,
        ...(params.category ? { category: params.category } : {}),
      },
      take: params.limit ?? 5,
      orderBy: { updatedAt: 'desc' },
    })
  },
}
```

### 3.2 Integration with Feature Generator Graph

The generator graph's `intake` node currently creates a `Feature` record. With abstraction:

```typescript
// In intake() node — MODIFIED flow:
async function intake(state: FeatureGeneratorState): Promise<Partial<FeatureGeneratorState>> {
  // ... existing prompt resolution ...

  // NEW: Create or find FeatureSpec (abstract)
  let featureSpecId = state.featureSpecId
  if (!featureSpecId && state.intakeData) {
    const spec = await featureSpecService.findOrCreate({
      name: state.intakeData.name,
      featureId: state.intakeData.featureId,
      description: state.intakeData.description,
      category: state.intakeData.category,
      tags: state.intakeData.tags,
      scope: state.intakeData.scope,
      tenantId: state.tenantId,
      userId: state.userId,
    })
    featureSpecId = spec.id

    // Check if we already have an implementation for this tech stack
    const hasImpl = await featureSpecService.hasImplementation(
      spec.id, state.intakeData.techStack,
    )
    if (hasImpl) {
      // Can offer to clone/adapt existing implementation instead of regenerating
      // This is a UX decision — for now, proceed with new generation
    }
  }

  // Create FeatureImplementation record (concrete)
  let featureDbId = state.featureDbId
  if (!featureDbId && state.intakeData && featureSpecId) {
    const impl = await prisma.featureImplementation.create({
      data: {
        featureSpecId,
        techStack: state.intakeData.techStack as Prisma.InputJsonValue,
        techStackKey: techStackKey(state.intakeData.techStack),
        tenantId: state.tenantId,
        createdBy: state.userId,
      },
    })
    featureDbId = impl.id
  }

  return {
    messages: [response],
    featureDbId,
    featureSpecId,
    phase: 'clarify',
  }
}
```

```typescript
  tenantId,
  category,
  limit: 5,

// For each matching spec, find the BEST implementation
// (highest quality, closest tech stack)

  const impls = await featureSpecService.getImplementations(spec.id)
  const bestImpl = impls.find(i => i.techStackKey === targetKey)
    ?? impls.sort((a, b) => b.quality - a.quality)[0]
  // Use bestImpl as reference
}
```

### 4.2 Cross-Stack Adaptation Pipeline

```

  │     Files: 15 overlay files
  │
  └─▶ [REQUESTED] Implementation (React + FastAPI)
        │
        ├─ Step 1: Use Vue+Express implementation as reference
        ├─ Step 2: Apply framework adaptation rules
```

## 6. Migration Strategy

### Phase 1: Non-Breaking (Sprint 1)

1. Create Prisma migration for `FeatureSpec` and `FeatureImplementation`
2. Add `featureSpecId` optional column to existing `Feature` model
3. Create `feature-spec.service.ts`
4. Add `featureSpecId` to generator state
5. Modify `intake()` to create FeatureSpec alongside Feature

### Phase 2: Backfill (Sprint 2)

1. Write backfill script: for each existing `Feature`, create corresponding `FeatureSpec`
2. Link `Feature.featureSpecId` to the new FeatureSpec
3. Index FeatureSpec descriptions in Qdrant for semantic search
4. Update RAG retrieval to search FeatureSpec first

### Phase 3: Multi-Stack (Sprint 3)

1. Update generator to use `FeatureImplementation` instead of `Feature` for file storage
2. Add "generate for different stack" workflow in frontend
3. Implement cross-stack reference in plan/generate nodes
4. Update template builder to reference FeatureSpec (not Feature) for composition
