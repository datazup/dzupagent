## 2. Current Implementation Snapshot

`@dzupagent/codegen` is a broad, modular package with strong internal decomposition. The implementation includes:
- VFS stack (`VirtualFS`, copy-on-write forks, diff/patch engine, snapshots, checkpoint manager).
- Generation stack (`CodeGenService`, incremental edit planner, test-spec generator).
- Sandbox stack (mock, Docker, E2B, Fly, WASM, K8s, hardening, pool, audit, volumes).
- Quality and guardrails (`QualityScorer`, convention gate, guardrail engine/rules, import/contract checks).
- Pipeline and orchestration (`GenPipelineBuilder`, `PipelineExecutor`, phase conditions, escalation).
- Developer workflow utilities (git tools, repo map, code review, CI failure routing/fix loop, PR state machine).
- Correction and learning loop (`SelfCorrectionLoop`, reflection node, lesson extractor).

Key strengths:
- Large and healthy test footprint with focused unit tests across most subsystems.
- Clear package segmentation and extensive public exports.
- Good use of pure functions for analyzers/validators, improving testability.
- Strong optionality and pluggability (sandbox providers, quality dimensions, guardrail rules, role/priority matrices).

## 3. Gap Analysis (Severity-Ranked)

## High

1. Sandbox lifecycle contract mismatch breaks `WorkspaceRunner` with cloud providers.
- Evidence:
  - `WorkspaceRunner` uploads before execute: `packages/codegen/src/vfs/workspace-runner.ts:74-77`
  - E2B `uploadFiles` requires initialized sandbox: `packages/codegen/src/sandbox/e2b-sandbox.ts:60-63`
  - Fly `uploadFiles` requires initialized machine: `packages/codegen/src/sandbox/fly-sandbox.ts:73-75`
- Impact:
  - `WorkspaceRunner.run()` can fail immediately for E2B/Fly when `uploadFiles()` is first call.
  - Provider interchangeability promised by `SandboxProtocol` is not fully honored.
- Recommendation:
  - Introduce a provider-agnostic `ensureReady()` phase in protocol or guarantee lazy init inside `uploadFiles()`/`downloadFiles()` in all providers.
  - Add conformance tests that run the same behavior contract across all sandbox providers.

2. Public documentation/API drift is material and user-facing.
- Evidence:
  - README quick-start uses old constructor/params: `packages/codegen/README.md:49-54`
  - Actual constructor/params differ (`ModelRegistry`, `filePath`, `purpose`): `packages/codegen/src/generation/code-gen-service.ts:24-35`
  - README claims `createWriteFileTool(vfs)` while function is zero-arg and non-mutating: `packages/codegen/README.md:141`, `packages/codegen/src/tools/write-file.tool.ts:7-17`
  - README autogenerated metadata stale (`Test Files | 9`), actual test files are 39: `packages/codegen/README.md:12`
  - README internal dependency list omits `@dzupagent/agent`: `packages/codegen/README.md:13`, `packages/codegen/package.json:21-24`
- Impact:
  - Higher integration friction and false onboarding paths for consumers.
  - Increased maintenance and support load due to incorrect usage examples.
- Recommendation:
  - Regenerate and validate README/API snippets from source signatures in CI.
  - Add docs contract test to fail when examples drift from exported API.

## Medium

3. Token budget manager can exceed configured budget.
- Evidence:
  - On overflow, `selectFiles()` still appends summary unconditionally in multiple branches: `packages/codegen/src/context/token-budget.ts:300-315`, `:317-320`.
- Impact:
  - Prompt size can exceed model limits unexpectedly.
  - Deterministic context sizing is not guaranteed.
- Recommendation:
  - Enforce a hard budget cap with drop policy (or truncation policy).
  - Return metadata about dropped files and overflow reason.

4. Pipeline timeout does not cancel underlying phase execution.
- Evidence:
  - `withTimeout()` races promise but does not propagate cancellation to `phase.execute`: `packages/codegen/src/pipeline/pipeline-executor.ts:103-123`.
- Impact:
  - Timed-out tasks may continue running in background, causing side effects/resource leaks.
  - Retry logic can overlap with still-running previous attempt.
- Recommendation:
  - Extend phase contract with abort signal (`execute(state, signal)`), wire `AbortController` through execution stack.
  - Ensure provider code (sandbox/tooling) honors cancellation.

5. Docker sandbox filesystem/mount settings are internally inconsistent for mutable workflows.
- Evidence:
  - Container is read-only and bind-mounts `/work` read-only while also mounting tmpfs at `/work`: `packages/codegen/src/sandbox/docker-sandbox.ts:95-102`.
- Impact:
  - Ambiguous runtime behavior depending on Docker mount precedence.
  - `syncBack`/edit workflows are at risk of being ineffective or brittle.
- Recommendation:
  - Split modes explicitly: `validation-only` (read-only) vs `mutable-workspace` (controlled write mount).
  - Add integration test against real Docker daemon for file write + download roundtrip.

6. K8s sandbox timeout reporting is post-hoc and does not enforce command timeout.
- Evidence:
  - `timedOut` is computed after `exec` returns via wall-clock comparison: `packages/codegen/src/sandbox/k8s/k8s-sandbox.ts:76-90`.
- Impact:
  - Long-running commands are not actively interrupted.
  - Timeout semantics are inconsistent with other providers.
- Recommendation:
  - Push timeout into client exec request and/or wrap in abortable call.
  - Standardize timeout semantics across providers through protocol tests.

7. Import validation is duplicated with two incompatible APIs and issue models.
- Evidence:
  - `validation/import-validator.ts` (`VirtualFS`, `errors`): `packages/codegen/src/validation/import-validator.ts:9-49`
  - `quality/import-validator.ts` (`Record|Map`, `issues`, cycle/self checks): `packages/codegen/src/quality/import-validator.ts:9-114`
  - Both exported from package root under similar names: `packages/codegen/src/index.ts:158-187`
- Impact:
  - Consumer confusion and fragmented behavior.
  - Higher maintenance burden and inconsistent fixes.
- Recommendation:
  - Consolidate into one import validation module with optional checks (`resolve`, `self`, `cycle`), unified result type.

8. `hasTests` quality dimension can undercount valid tests in common layouts.
- Evidence:
  - Coverage matching only checks basename variants next to source path: `packages/codegen/src/quality/quality-dimensions.ts:147-152`.
- Impact:
  - False warnings for projects using mirrored `__tests__` trees or alternate test folder conventions.
- Recommendation:
  - Add configurable test-path resolvers (co-located, mirrored, framework-specific conventions).

## Low

9. Code review engine applies rules line-by-line, reducing effectiveness of multiline rules.
- Evidence:
  - Line-by-line scanning: `packages/codegen/src/review/code-reviewer.ts:65-92`
  - Includes multiline-intent rule (`PERF-001`) with `[\s\S]*?`: `packages/codegen/src/review/review-rules.ts:33`
- Impact:
  - Missed detections for cross-line patterns and control-flow context.
- Recommendation:
  - Add file-level and AST-level rule execution mode for selected rules.

10. Integration realism is still limited for optional infra paths.
- Evidence:
  - K8s operator suites can be skipped when operator module unavailable: `packages/codegen/src/__tests__/k8s-sandbox.test.ts:34-40`.
- Impact:
  - Some critical runtime paths remain less validated in CI.
- Recommendation:
  - Add opt-in integration matrix (Docker/K8s/E2B/Fly) in dedicated CI jobs with feature flags.

## 4. Architectural and Product Opportunities

The package is already feature-rich. The next gains come from reliability contracts, observability, and stronger cross-module coherence.

Proposed feature set (prioritized):

1. Sandbox Provider Conformance Suite.
- Add shared behavioral tests for all providers covering init lifecycle, file roundtrip, timeout, cancellation, cleanup, and error mapping.
- Outcome: provider interchangeability becomes real and measurable.

2. Unified Execution Session API.
- Introduce `SandboxSession` abstraction with explicit lifecycle: `init -> upload -> exec -> download -> cleanup`.
- Outcome: fixes lifecycle mismatch and clarifies protocol semantics.

3. Strict Token Budget Planner V2.
- Hard cap with deterministic inclusion policy.
- Optional tokenizer-backed accounting per model family.
- Outcome: stable prompt sizing and better cost/runtime control.

4. Contract Validator V2 (Path Templates + Params).
- Normalize dynamic segments (`/users/:id`, `/users/{id}`, `/users/123`) to canonical template.
- Optionally parse OpenAPI or typed route definitions.
- Outcome: fewer false positives and better frontend/backend coherence checks.

5. API/Docs Drift Guard.
- CI step that compiles examples from README and validates exported signatures.
- Outcome: onboarding docs remain trustworthy.

6. Pipeline Abortability and Side-Effect Isolation.
- Abort-signal propagation across pipeline phases and sandbox/tool execution.
- Outcome: safer retries and no hidden background work after timeout.

7. Rule Engine Dual Mode (Regex + AST).
- Keep fast regex baseline; add AST enrichments for precision-critical rules.
- Outcome: higher signal for review and guardrail checks.

8. Migration Planner Confidence Scoring.
- Add confidence score per migration step from detected constructs, not only file-count heuristics.
- Outcome: better migration risk estimation and planning quality.

9. Quality Dimension Plugin Registry.
- Runtime registration with labels, severity policy, and explainability metadata.
- Outcome: package consumers can tailor quality gates by domain.

10. Operational Telemetry Hooks.
- Structured events for generation, validation, retry, guardrail failures, sandbox actions, and token/cost usage.
- Outcome: easier production debugging and optimization loops.

## 5. Suggested Roadmap

Immediate (1-2 weeks):
- Fix sandbox lifecycle mismatch (`WorkspaceRunner` + E2B/Fly).
- Correct README/API drift and add docs contract checks.
- Make token budget enforcement strict.

Short term (2-6 weeks):
- Implement abort-signal execution path in pipeline and providers.
- Consolidate import validators into one unified API.
- Add provider conformance tests and Docker roundtrip integration test.

Mid term (6-10 weeks):
- Contract validator V2 with canonical path templates.
- Rule engine AST augmentations.
- Telemetry and observability hooks.

## 6. Final Assessment

Current implementation quality is strong for a large experimental package: robust module decomposition, broad capability set, and substantial tests.

Primary risks are not basic correctness failures but integration contract mismatches and API/documentation coherence issues.

If the high/medium items above are addressed, `@dzupagent/codegen` can move from “feature-rich experimental” to “reliable platform module” suitable for broader internal and external adoption.
